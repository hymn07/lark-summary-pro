# 02.5 — 业务逻辑设计规范

> 作用：数据库 Schema 定义了「存什么」，本文档定义「处理流程怎么做」。
> 这是产品工程师最核心的设计能力——把业务流程变成可执行的代码结构。

---

## 一、硬性规则（MUST）

### 1.1 每个核心业务对象必须有状态机

不需要画完整 UML，但必须明确定义：

```typescript
// 审批场景
pending → approved  // 通过
pending → rejected  // 拒绝
pending → cancelled // 取消

// 报告场景
received → in_progress → completed
                        → blocked
                        → overdue

// 问题场景
open → investigating → resolved
                     → wont_fix
```

**规则**：
- 状态变更是后端的责任，前端只负责触发
- 每个状态变更必须有 timeline/audit log 记录
- 不允许直接从任意状态跳到任意状态——定义好允许的转换路径

### 1.2 每个业务对象必须有初始状态

根据分类和提取结果确定初始状态，不能全部默认同一个值：

```typescript
function determineInitialStatus(type, extractedFields): string {
  switch (type) {
    case "approval": return "pending";
    case "report":   return extractedFields.overallStatus === "blocked" ? "blocked" : "in_progress";
    case "issue":    return "open";
    default:         return "pending";
  }
}
```

### 1.3 核心处理流水线必须有明确的步骤

每个步骤：输入 → 处理 → 输出，错误了怎么办：

```
新数据到达
  ↓
Step 1: 预处理（去重、清洗、格式标准化）
  ↓ 失败 → 标记为「预处理失败」，记录原始数据，等人工处理
Step 2: AI 理解（分类 + 抽取 + 摘要）
  ↓ 失败 → 降级：创建未分析实体，标记 requiresReview=true
Step 3: 匹配/聚合（新数据关联到已有数据还是创建新的？）
  ↓
Step 4: 写入（实体 + 关联数据 + timeline）
  ↓
Step 5: 后处理（通知、AI 建议、打标签）
  ↓ 失败 → 非关键路径，catch error 不影响主流程
```

## 二、强烈建议（SHOULD）

### 2.1 聚合匹配策略：从严到宽

当一条新数据到来时，如何判断它是「新的」还是「已有记录的后续」？

**三层匹配模式**：

```
层级1（精确匹配）：唯一标识符
  → 如邮件 thread ID、订单号、用户 ID
  → 这一层命中了就是确定匹配

层级2（启发式匹配）：多个字段组合 + 时间窗口
  → 如：同发送者 + 相似标题 + 30 天内
  → 标题先去前缀（Re:/Fwd:/回复：）再做匹配

层级3（AI 语义匹配）：用 LLM 判断语义相关性
  → 在更大时间窗口（60-90天）内
  → 把候选记录信息喂给 LLM，让它判断是否同一事物
  → 适合标题表述不同但本质同一事务的情况
```

**为什么三层？**
- 层级1 是确定性的，零成本
- 层级2 是启发式的，低成本
- 层级3 是 AI 的，高成本但高召回

每一层消费递增，所以从最便宜的开始。

### 2.2 字段合并策略

当后续数据更新已有记录时，不能简单覆盖：

```typescript
function mergeFields(existing: Record<string, any>, incoming: Record<string, any>) {
  const merged = { ...existing };  // 保留所有旧值

  for (const [key, value] of Object.entries(incoming)) {
    // null/undefined → 不覆盖（新数据没提到的字段保持旧值）
    if (value === null || value === undefined) continue;
    // 空数组 → 不覆盖（新数据没有不代表旧的没有）
    if (Array.isArray(value) && value.length === 0) continue;
    // 空字符串 → 不覆盖（除非旧值为空）
    if (value === "" && existing[key] !== "" && existing[key] !== undefined) continue;
    // 其他情况 → 用新值覆盖
    merged[key] = value;
  }

  return merged;
}
```

**核心思想**：只有新数据明确提供了的信息才更新。空值不代表「没有」，只代表「这次没提到」。

### 2.3 状态机实现模式

```typescript
// 定义允许的状态转换
const ALLOWED_TRANSITIONS: Record<string, string[]> = {
  pending:    ["approved", "rejected", "cancelled"],
  approved:   [],
  rejected:   ["pending"],  // 重新提交
  cancelled:  [],
};

function validateTransition(from: string, to: string): boolean {
  const allowed = ALLOWED_TRANSITIONS[from];
  if (!allowed) return false;
  return allowed.includes(to);
}

async function updateStatus(entityId: string, newStatus: string, userId: string, reason?: string) {
  const entity = await db.entity.findUnique({ where: { id: entityId } });

  if (!validateTransition(entity.status, newStatus)) {
    throw new Error(`Cannot transition from ${entity.status} to ${newStatus}`);
  }

  // 在一个事务中完成：更新状态 + 记录变更日志
  return db.$transaction([
    db.entity.update({ where: { id: entityId }, data: { status: newStatus } }),
    db.stageTransitionLog.create({
      data: {
        entityId,
        fromStage: entity.status,
        toStage: newStatus,
        changedBy: userId,
        reason,
      },
    }),
  ]);
}
```

**关键**：状态转换验证必须在后端，前端只负责调 API。

### 2.4 后续数据到达时的处理策略

当一条新数据被匹配到已有记录时（如后续邮件追加到同一个 Entity），不只是简单关联：

```
后续数据到达
  ↓
1. 追加数据记录（如 EmailThread 关联到 Entity）
2. 创建 TimelineEvent："收到来自 xx 的后续信息"
3. 如果启用 AI：
   a. 重新运行 AI 字段抽取（只基于新数据）
   b. 合并抽取结果到已有字段
   c. 重新生成 AI 摘要
   d. 生成 AI 建议（下一步该做什么）
4. 计算进度变化（如果字段中包含 progress 信息）
5. 记录所有变更
```

### 2.5 降级路径

AI 功能不是全年无休的。每个 AI 依赖的路径都要有降级方案：

```typescript
// 分类降级
if (!hasAiKey) {
  entity.type = "uncategorized";
  entity.requiresReview = true;
}

// 抽取降级
if (classificationFailed) {
  entity.extractedFields = {};  // 空字段，用户手动填
}

// AI 摘要降级
if (summaryFailed) {
  entity.aiSummary = content.slice(0, 200);  // 取原文前 200 字
}

// 给 Agent 的信号
result.requiresManualReview = confidence < 0.7 || !hasAiKey;
```

## 三、建议（MAY）

### 3.1 处理流程可拆分为多个函数

不要把 600 行的处理函数写在一个文件里。拆成独立的步骤：

```
processor.ts        → 主入口，编排步骤
classifier.ts       → 步骤 1：分类
extractor.ts        → 步骤 2：字段抽取
summarizer.ts       → 步骤 3：摘要
matcher.ts          → 步骤 4：聚合匹配
tagger.ts           → 步骤 5：自动标签
```

### 3.2 异步非关键步骤

耗时操作（AI 分析、通知发送）可以异步处理：

```typescript
// 关键路径：同步完成，用户需要看到结果
await createEntity(data);
await createTimeline(data);

// 非关键路径：异步处理，失败不影响主流程
aiAnalysis(data).catch(logError);
sendNotification(data).catch(logError);
recordMemory(data).catch(logError);
```

### 3.3 考虑幂等性

基于内容哈希（contentHash）做去重，防止同一数据被处理两次：

```typescript
const hash = createHash(data.content);
const existing = await db.entity.findUnique({ where: { contentHash: hash } });
if (existing) return existing;  // 已处理过
```

## 四、从实际项目中提炼的 Insight

### 4.1 聚合匹配的三层设计是最容易被 AI 漏掉的

如果你只告诉 AI「新邮件来了就创建 Entity」，它会做一个最简单的 version：每封邮件创建一个 Entity。不会想到三层匹配。

**给 AI 的指令中必须强调**：新数据到来时，先判断是不是已有记录的后续，再决定创建还是追加。匹配策略：精确 ID → 启发式规则 → AI 语义匹配。

### 4.2 字段合并的「空值不覆盖」是无数 bug 的根源

很多系统的问题是：后续数据只提到了 A，没提到 B——如果直接覆盖，B 就被清空了。这个原则看起来简单，但没有经验的开发者几乎一定会写错。

### 4.3 状态机的严格性决定产品的质量

如果前端可以直接改状态、没有转换校验、没有变更日志——那这只是一个 demo。真正的产品在后端强制状态转换的合法性。

### 4.4 非关键步骤异步化

不是每个处理步骤都需要同步完成。AI 分析、标签生成、通知发送、记忆更新——这些失败了不应该阻碍主流程。`catch(() => {})` 和 `async + catch(logError)` 是你的朋友。

## 五、快速对照：处理流水线需要定义什么

开始实现一个业务功能前，填这张表：

```
| 步骤 | 输入 | 输出 | 成功了 | 失败了 |
|------|------|------|--------|--------|
| 预处理 | 原始数据 | 标准化数据 | 继续下一步 | 标记待人工处理 |
| AI 理解 | 标准化数据 | 分类+字段+摘要 | 继续下一步 | 未分析实体 |
| 匹配 | 理解结果 | 已有ID或null | 追加或新建 | 新建（安全兜底） |
| 写入 | 数据+匹配结果 | 数据库记录 | 继续下一步 | 回滚+报错 |
| 后处理 | 新记录 | 通知+标签+记忆 | 无所谓 | catch忽略 |
```
