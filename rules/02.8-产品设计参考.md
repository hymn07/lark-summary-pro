# 02.8 — 产品设计参考

> 作用：给 AI 提供具体的产品设计模式参考，让 AI 知道「好的设计长什么样」。
> 本文档不规定死板的模板，而是提供经过验证的设计模式和具体例子。

---

## 一、怎么用这个文档

当你说「我要一个事项列表页」「我要一个详情页」时，AI 会自己设计。但如果你想让设计更接近某种风格，你可以：

```
"列表页用 Inbox 模式" → AI 理解你想要的布局
"详情页加 Agent 面板" → AI 理解你要右侧 AI 辅助
"状态用颜色+图标区分" → AI 理解视觉语言
```

**但即使你不说这些关键词，AI 读了本文档后也会默认采用这些模式。**
本文档是给 AI 的「设计教育」，读完它会让 AI 的设计品位提升。

---

## 二、核心页面模式

### 模式 1：Inbox 模式（结构化列表）

**适用场景**：用户需要快速浏览并分类处理大量条目。

```
┌──────────────────────────────────────────────────────┐
│  [搜索框]                    [分类标签] [筛选] [排序]  │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │ 📋 [审批] 市场部 Q2 广告预算申请  [待处理]    │    │
│  │    金额 ¥50,000 · 申请人 李明 · 3小时前      │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │ 📊 [周报] 技术团队 W13 进展汇报  [进行中]     │    │
│  │    阻塞项: API不稳定 · 王芳 · 昨天           │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │ 🐛 [问题] 线上支付回调失败率上升    [开放]    │    │
│  │    影响: 约5%用户 · 紧急 · 2小时前           │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**设计要点**：
- 每条不是「标题+正文片段」的邮件列表，而是「类型标签 + 关键字段 + 状态」的结构化卡片
- 分类标签用颜色+图标区分，用户可以一键筛选
- 每张卡片 3-4 行信息，让用户一眼判断优先级
- 关键信息不是截断原文，而是 AI 提取后的结构化摘要

### 模式 2：Entity Workspace（详情页 + AI 面板）

**适用场景**：用户深入处理一个具体事项。

```
┌────────────────────────────────┬───────────────────────┐
│                                │                       │
│  ← 返回列表                    │  🤖 AI Assistant      │
│                                │                       │
│  市场部 Q2 广告预算申请         │  "这是张伟申请的      │
│  [审批] [待处理]               │   Q2广告预算¥50000。  │
│                                │   建议通过。"          │
│  ┌─────────────────────────┐   │                       │
│  │ AI 摘要                 │   │  💡 建议：            │
│  │ 张伟申请市场部Q2广告    │   │  - 确认预算不超       │
│  │ 投放预算50000元。       │   │  - 标注关键日期       │
│  │ 投放平台: 微信+抖音     │   │  [执行建议]           │
│  └─────────────────────────┘   │                       │
│                                │  💬 对话...            │
│  📋 提取字段                   │                       │
│  金额: ¥50,000                 │                       │
│  申请人: 张伟                  │                       │
│  部门: 市场部                  │                       │
│  截止日期: 3月31日             │                       │
│                                │                       │
│  📅 Timeline                   │                       │
│  ● 14:30 张伟发来审批邮件     │                       │
│  ● 14:32 AI 提取关键信息      │                       │
│  ○ 待操作: 审批               │                       │
│                                │                       │
│  [✓ 通过] [✗ 拒绝] [💬 评论]  │                       │
│                                │                       │
└────────────────────────────────┴───────────────────────┘
```

**设计要点**：
- 左侧 70%：详情内容（摘要 → 字段 → 时间线 → 操作）
- 右侧 30%：AI 面板（上下文感知的 AI 助手，自动总结当前事项，给出建议）
- Timeline（时间线）按时间顺序展示所有相关事件，不同类型用不同图标
- 操作按钮（通过/拒绝）是明确的行为召唤，不是隐藏在下拉菜单里

### 模式 3：Dashboard 模式（多区域仪表盘）

**适用场景**：用户打开应用时看到的第一页，汇总所有需要关注的信息。

```
┌──────────────────────────────────────────────────────┐
│  Good morning, [Name]!                              │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌── 待处理 (3) ─────────────────────────────────┐  │
│  │  [审批] Q2预算申请 · ¥50,000 · 3小时前         │  │
│  │  [问题] 支付异常 · 紧急 · 2小时前               │  │
│  │  [审批] 服务器采购 · ¥35,000 · 昨天             │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
│  ┌── 进行中 (5) ─────────────────────────────────┐  │
│  │  [周报] 技术团队 W13 · 进行中                   │  │
│  │  [周报] 市场部月度 · 有阻塞                     │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
│  ┌── AI 洞察 ────────────────────────────────────┐  │
│  │  "本周有2个审批逾期，建议今天处理"               │  │
│  │  "市场部周报连续2周提到预算不足"                 │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**设计要点**：
- 分区展示：每个区域用标题+边框区分，有独立的「重要性」
- 待处理放最上面，进行中其次，AI 洞察做锦上添花
- 每个分区的数量（3）（5）是用户决策的关键信息
- AI 洞察不是随机生成的内容——是从数据中提取的真实信号

## 三、视觉语言设计

### 3.1 状态颜色系统

所有状态都应该用**颜色 + 图标**双通道区分（考虑色盲用户）：

| 状态类型 | 颜色 | 图标 | 使用场景 |
|---------|------|------|---------|
| 成功/完成 | 绿色 | CheckCircle | approved, completed |
| 进行中/活跃 | 蓝色 | Clock | in_progress, active |
| 警告/待处理 | 黄色/橙色 | AlertCircle | pending, warning |
| 错误/阻塞 | 红色 | XCircle, AlertTriangle | rejected, blocked, error |
| 默认/未知 | 灰色 | Inbox, Info | received, unknown |

### 3.2 预警等级

当数据需要标注风险或优先级时：

| 等级 | 颜色 | 含义 |
|------|------|------|
| 🔴 红色/紧急 | 高风险，需要立即处理 |
| 🟡 黄色/警告 | 需要关注，但不必立即行动 |
| 🟢 绿色/正常 | 一切正常，无负面信号 |
| ⚪ 灰色/静默 | 无数据或无法判断 |

### 3.3 图标使用原则

- 每个分类、每个状态、每种操作都有一个固定的图标（用 lucide-react）
- 图标是用户快速扫描的锚点，不是装饰
- 同一图标在全局代表同一个含义（别这里 CheckCircle 是「完成」那里又是「审批」）

## 四、导航设计

### 4.1 侧边栏分区

按用户心智模型分区，而不是按功能类型分区：

```
── 业务核心 ──    ← 用户每天用的，放最上面
🏠 仪表盘
📋 收件箱
✅ 审批中心

── 工具 ──        ← 偶尔用的
📊 数据看板

── 设置 ──        ← 很少用的，放最下面
⚙️ 通用设置
👥 成员管理
```

**设计要点**：
- 分区用细线或间距分隔，不要混成一团
- 「业务核心」放最上面（用户最常访问）
- 「设置」「管理」放最下面或底部
- 每个菜单项带一个小图标
- 当前激活的页面高亮

### 4.2 顶部栏

```
[Logo] [组织选择器 ▼]                    [通知🔔] [头像 ▼]
```

组织选择器：如果产品是多租户的，顶部栏有组织/工作区切换器。

## 五、交互模式

### 5.1 操作流程（标准模式）

```
用户点击操作按钮
  → 弹出确认对话框
    → 确认 → 按钮 loading → API 调用
      → 成功 → toast 提示 → 自动关闭对话框 → 刷新列表
      → 失败 → toast 提示错误 → 对话框保持打开（可重试）
    → 取消 → 关闭对话框
```

**为什么这样设计**：
- 确认对话框防止误操作
- loading 状态防止重复点击
- 成功后自动刷新——用户不需要手动刷新页面
- 失败后对话框不关闭——用户可以重试而不重新填写

### 5.2 列表页交互

```
1. 页面加载 → skeleton 占位（不是转圈）
2. 数据加载成功 → 展示列表
3. 数据为空 → 显示空状态引导（"还没有任何事项，连接邮箱开始" + 操作按钮）
4. 加载失败 → 显示错误提示 + 重试按钮
```

**Skeleton 的用法**：不是一个大转圈。是灰色块占位，模拟真实内容的形状。

### 5.3 Toast 使用

```
成功操作 → 绿色 toast，在右上角出现，3秒后自动消失
失败操作 → 红色 toast，显示错误原因，5秒后自动消失
重要提醒 → 蓝色 toast
```

## 六、列表页常用布局模式

### 6.1 布局选择指南

| 数据特征 | 推荐布局 |
|---------|---------|
| 每条数据有 3-6 个结构化字段，需要快速扫描 | 紧凑卡片列表 |
| 每条数据信息很多（10+字段），需要详细比较 | 表格 |
| 数据有层级关系（文件夹/项目） | 树形侧栏 + 右侧内容 |
| 需要同时看列表和详情 | 左右分栏（列表 | 详情） |

### 6.2 卡片列表模式

每张卡片的内容结构：
```
[类型图标] [分类标签] [标题]
[关键字段1] · [关键字段2] · [关键字段3]
[相对时间] · [来源/发件人]
```

不超过 3 行。用户扫描一屏最多看到 5-6 张卡片。

## 七、页面组件结构

### 7.1 组件命名

```
业务组件放 modules/[feature]/components/
  EntityList.tsx      → 列表组件
  EntityDetail.tsx    → 详情组件
  EntityCard.tsx      → 卡片组件
  EntityTimeline.tsx  → 时间线组件
  ApprovalActions.tsx → 操作组件

通用UI组件放 packages/ui/components/
  Button, Card, Dialog, Badge, Tabs, etc.
```

### 7.2 每个页面必须处理的三种状态

```typescript
// 加载态
if (isLoading) return <EntitySkeleton />;   // 不是 <Spinner />

// 错误态
if (error) return <ErrorCard message={error.message} onRetry={refetch} />;

// 空态
if (!data?.length) return <EmptyState
  icon={InboxIcon}
  title="还没有任何事项"
  description="连接你的邮箱后，系统会自动同步邮件并分类"
  action={<Button>连接邮箱 →</Button>}
/>;

// 正常态
return <EntityList data={data} />;
```

## 八、把这些告诉 AI

### 你可以说（关键词触发）

```
"详情页用 Entity Workspace 模式"     → 左右分栏，右侧 AI 面板
"列表页用 Inbox 模式"                → 结构化卡片，分类筛选
"首页用 Dashboard 模式"              → 多区域汇总
"状态用颜色+图标"                    → badge 系统
"加 timeline"                        → 时间线组件
"加 Agent 面板"                      → 右侧 AI 对话
"加 AI 洞察区"                        → AI 生成的信号汇总
```

### 也可以直接描述

```
"详情页做成左右两栏，左边是内容，右边是一个AI助手面板，
 AI能自动总结当前这个事项并给出建议"
```

```
"详情页参考 Notion 的页面布局，但右侧加一个 ChatGPT 那样的对话面板"
"列表页参考 Gmail 的分类标签方式"
```

## 九、核心设计原则（AI 必须遵循）

1. **结构化 > 原始文本**：UI 展示的是 AI 提取后的结构化信息，不是原始输入
2. **人在回路**：AI 给建议，人做决定。操作按钮必须明确、可见
3. **三种状态缺一不可**：loading、error、empty，每个组件都要处理
4. **操作要有反馈**：确认 → loading → toast → 自动刷新
5. **导航分区**：按用户心智模型分组，最常用的放最上面
6. **颜色不是唯一的信息**：状态用「颜色 + 图标 + 文字」三通道
7. **不要让用户猜**：每个按钮的文案说清楚会发生什么（「通过审批」>「确认」）
