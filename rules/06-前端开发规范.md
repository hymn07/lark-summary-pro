> 作用：写出统一的、可维护的、用户友好的前端代码

## 一、硬性规则（MUST）

> ⚠️ 规则冲突提醒：如果以下规则与你的实际需求有任何冲突，AI 必须主动指出并建议替代方案（参见 00-AI协作原则.md）。

### 1.1 组件分层

```
apps/web/modules/
├── saas/          # SaaS 共享组件（布局、导航、组织选择器）
├── marketing/     # 营销页面组件（首页、定价等）
├── shared/        # 跨场景通用组件
└── [feature]/     # 业务功能组件

packages/ui/
└── components/    # shadcn/ui 基础组件
```

- 业务组件放 `modules/[feature]/`
- 通用 UI 组件放 `packages/ui/`
- 不跨功能复用业务组件

### 1.2 Server Component 优先

```
✅ 默认写 Server Component（async，直接读数据库，零客户端 JS）
✅ 只在需要交互时加 "use client"
✅ 数据获取用 TanStack Query（在 client component 中）
✅ 表单用 react-hook-form + zod

❌ 禁止在 Server Component 之外直接调数据库
❌ 禁止用 useEffect 做数据获取
❌ 禁止在组件中直接调 Prisma client
```

### 1.3 状态管理铁律

```
Server State（来自数据库的数据）
  → TanStack Query（自动缓存、失效、重取、乐观更新）

Client State（纯 UI 状态）
  → useState

URL 参数状态（分页、筛选、搜索词）
  → nuqs（URL search params 双向绑定）

表单状态（输入、校验、提交）
  → react-hook-form + zod

❌ 不要用 Zustand / Redux / MobX
❌ 不要在多个组件间手动同步状态
❌ 不要用 Context 管理 Server State
```

## 二、强烈建议（SHOULD）

### 2.1 路由组织模式

```
app/
├── (marketing)/           # 公开页面（SEO 友好）
│   └── [locale]/
│       ├── (home)/        # 首页
│       └── pricing/       # 定价
│
├── (saas)/                # 需登录
│   └── app/
│       ├── [organizationSlug]/
│       │   ├── page.tsx       # Dashboard
│       │   ├── [feature]/     # 业务页面
│       │   └── settings/      # 组织设置
│       └── (account)/         # 个人账户
│           └── settings/
│
├── auth/                  # 登录/注册/验证
└── api/                   # API 路由
```

### 2.2 每个数据展示组件必须处理三态

```typescript
function EntityList() {
  const { data, isLoading, error, refetch } = useQuery(...)

  if (isLoading)  return <Skeleton />       // 加载态
  if (error)      return <ErrorCard onRetry={refetch} />  // 错误态
  if (!data?.length) return <EmptyCard />    // 空态

  return data.map(item => <ItemCard key={item.id} {...item} />)
}
```

### 2.3 操作交互模式

```typescript
// 标准操作流程
点击操作按钮
  → 弹出确认弹窗（Dialog）
  → 点击确认 → 按钮进入 loading 状态
  → API 调用成功 → 
    → toast 提示成功
    → 自动关闭弹窗
    → TanStack Query 自动刷新列表
  → API 调用失败 →
    → toast 提示错误原因
    → 弹窗不关闭（用户可重试）
```

### 2.4 侧边栏导航分区设计

```
将功能按心智模型分为区域：

── 📈 业务核心 ──
🏠 仪表盘
🎯 核心功能 A
📊 核心功能 B

── ⚙️ 账户设置 ──
👤 个人设置
🏢 组织管理

── 🔧 系统管理 ── [仅 Admin 可见]
🛠️ 管理面板
```

## 三、建议（MAY）

### 3.1 组件设计原则

- 列表页：搜索栏 + 筛选器 + 卡片/表格 + 分页
- 详情页：信息区 + timeline + 操作区 + AI 辅助区
- 复杂页面用三栏布局：列表 | 详情 | AI 侧栏

### 3.2 CSS 使用

- 优先用 Tailwind utility class
- 复用模式抽成 `@apply` 或组件
- 不引入 CSS-in-JS
- 响应式断点：sm(640) / md(768) / lg(1024) / xl(1280)

## 四、从实际项目中提炼的 Insight

### 4.1 AI 前端不只是聊天框

AI-native 产品的前端模式：
- **Inbox 模式**：列表不按时间排，按 AI 分类/优先级排
- **结构化卡片**：展示 AI 提取的关键字段，不是原始文本
- **Timeline 模式**：每个业务对象旁边是 AI 生成的摘要 + 事件时间线
- **操作侧栏**：AI 建议下一步操作，用户一键确认

### 4.2 类型安全的完整链路

```
Prisma Schema → Zod → oRPC types → TanStack Query → 组件 props

改 Prisma 一个字段 → 全链路编译报错 → 不会漏改
```

### 4.3 Landing Page 的设计

Landing Page 不是「好看就行」，要能讲清楚三件事：
1. 这是什么产品？（Hero 区一句话 + 一个图）
2. 能解决什么问题？（3 个场景卡片）
3. 怎么开始用？（CTA 按钮 + 定价）

不要堆砌技术特性。用场景说话，让用户 10 秒内对号入座。

### 4.4 前端页面 = 固定组件 + 可配置 Skill

一个先进的设计模式：前端页面保持不变，用户定制化需求通过组合后端 Tool/Skill 实现。这样前端不会因为每个用户的定制需求而不断膨胀。

例：用户想要「每周自动筛选并推送汇总」→ 不需要前端新页面，而是在后台创建 Skill（定时触发 + 筛选 Tool + 推送 Tool 的组合）。
