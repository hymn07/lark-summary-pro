# 00 — 产品设计规范

> 来源：从实际 AI SaaS 产品中提炼的设计方法
> 作用：在写任何代码之前，先把产品定义清楚

---

## 一、硬性规则（MUST）

### 1.1 必须有「一句话定位」

格式：`以 [入口] 为起点的 [产品类型] —— [核心主张]`

```
✅ 正确:  以邮箱为入口的 AI 业务执行平台 —— 把邮件变成可被执行的工作流
✅ 正确:  面向投资团队的 AI 操作系统 —— 把全流程数字化并 AI 增强
❌ 错误:  一个很好用的 SaaS 工具
```

### 1.2 必须有产品分层架构图

```
数据入口层（邮箱/API/数据源）
       ↓
能力层（解析 + AI 理解 + 业务实体 + 状态管理）
       ↓
执行层（通知 / 更新外部系统 / 生成报告）
       ↓
展示层（Dashboard + 列表 + 详情 + 操作面板）
```

### 1.3 必须定义 MVP 范围（最多 3 个场景）

每个场景写清楚：触发条件 → 处理过程 → 执行结果

```markdown
#### 场景名：审批
- 触发：何时启动这个功能
- 过程：AI 做什么 → 系统做什么
- 结果：输出什么、谁看到、下一步是什么
```

### 1.4 必须有明确的禁止清单

```markdown
## ❌ MVP 阶段不做
- [列出所有不在范围内的功能]
```

### 1.5 必须定义用户角色

| 角色 | 核心需求 | 关键操作 |
|------|---------|---------|
| 主角色 1 | 核心价值 | 具体操作 |
| 主角色 2 | 核心价值 | 具体操作 |
| 管理员 | 配置管理 | 系统设置 |

## 二、强烈建议（SHOULD）

### 2.1 用「场景驱动」而不是「功能列表驱动」

```
❌ 错误方式: 我要做一个仪表盘、一个列表页、一个搜索功能
✅ 正确方式: 用户每天早上打开系统，看到今天需要处理的 5 件事，
   每件事都有 AI 提取的关键信息和推荐操作
```

思考每个页面时，问三个问题：
- 用户在这个页面想完成什么任务？
- AI 能给用户省掉哪一步？
- 这个页面的核心数据对象是谁？

### 2.2 先定义核心页面，再写代码

每个页面的标准描述模板：
```
页面名：[名称]
定位：一句话，做什么的
布局：(三栏 / 两栏 / 单页)
内容：
  - 顶部：搜索 + 筛选 + 操作
  - 左/中部：核心数据展示
  - 右/底部：AI 辅助 / 详情
```

### 2.3 用「减法思维」做 MVP

砍功能的判断标准：
> 去掉这个功能，用户还能完成核心任务吗？
> 能 → 砍掉
> 不能 → 保留

## 三、从实际项目中提炼的 Insight

### 3.1 「人在回路」是关键设计原则

AI 的判断不能自动执行，而是标记置信度，低置信度时让用户确认。

**设计模式**：
- AI 分类置信度 < 0.7 → 标记 `requiresReview: true`
- 用户手动确认 → 记录为训练信号
- 用户修改 AI 判断 → 记录偏好，下次更准确

### 3.2 场景深度 > 场景广度

一个完整场景的链路应该做到：
```
数据进入 → AI 分类/抽取 → 结构化存储 → 前端展示 → 用户操作 → 结果反馈 → 记录日志
```

比 5 个只做到 60% 的场景有价值得多。

### 3.3 结构化数据 > 自然语言展示

AI 的核心价值是把非结构化信息变成结构化数据。前端应该展示结构化的结果，而不是原始输入。

```
❌ 展示: "这是一封邮件..." → 用户需要自己读
✅ 展示: 标题 | 分类 | 关键字段 | 状态 | 操作按钮 → 用户 3 秒理解
```

### 3.4 产品分层要分离「入口层」和「能力层」

入口层（邮箱、API、数据源）和能力层（AI 理解、业务实体、执行）是两层。入口层可以扩展（今天接 Gmail，以后接飞书），但能力层不变。

这种分离让产品可以「多入口、单能力」，入口扩展不影响核心逻辑。
