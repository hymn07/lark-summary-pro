# 04 — AI 集成规范

> 作用：让 AI 功能可靠、可控、可降级

---

## 一、硬性规则（MUST）

> ⚠️ 规则冲突提醒：如果以下规则与你的实际需求有任何冲突，AI 必须主动指出并建议替代方案（参见 00-AI协作原则.md）。

### 1.1 所有 AI 输出必须用 Zod Schema 约束

```typescript
// ✅ 正确
import { generateObject } from "ai";
import { z } from "zod";

const schema = z.object({
  type: z.enum(["A", "B", "C"]),
  confidence: z.number().min(0).max(1),
  reasoning: z.string(),
});
const result = await generateObject({ model, schema, prompt });

// ❌ 错误
const result = await generateText({ model, prompt: "返回 JSON..." });
const parsed = JSON.parse(result.text); // 可能解析失败
```

### 1.2 必须有降级路径

AI 不可用时，系统不能崩溃：

```typescript
async function processWithAI(input: Input): Promise<Result> {
  const hasAiKey = !!getAIKey();

  if (!hasAiKey) {
    return createRawResult(input, { requiresManualReview: true });
  }

  try {
    const aiResult = await aiProcess(input);
    return { ...aiResult, requiresManualReview: aiResult.confidence < 0.7 };
  } catch (error) {
    console.error("AI failed:", error);
    return createRawResult(input, { requiresManualReview: true, error: "AI unavailable" });
  }
}
```

### 1.3 低置信度必须标记人工确认

```typescript
const REVIEW_THRESHOLD = 0.7;

if (result.confidence < REVIEW_THRESHOLD) {
  entity.requiresReview = true;
}
```

## 二、强烈建议（SHOULD）

### 2.1 模型分层策略

| 任务 | 模型级别 | 原因 |
|------|---------|------|
| 分类 | fast model | 任务简单，量大 |
| 抽取/结构化 | text model | 需要准确性 |
| 摘要 | fast model | 任务轻量 |
| 语义匹配 | text model | 需要理解能力 |
| 复杂分析 | reasoning model | 高价值任务 |

```typescript
export const AI_MODELS = {
  fast: process.env.AI_FAST_MODEL || "gpt-4o-mini",
  text: process.env.AI_TEXT_MODEL || "gpt-4o",
  reasoning: process.env.AI_REASONING_MODEL || "o3-mini",
};
```

### 2.2 Prompt 管理规范

- 统一存放（`packages/ai/prompts/` 或 Prompt 配置表）
- 每个 Prompt 有版本记录
- 关键 Prompt 有对应测试用例
- 分类用 structured prompt + Zod schema 双重约束

### 2.3 支持多 AI Provider

```typescript
// 优先级：Anthropic 原生 > OpenAI 兼容网关 > 官方 OpenAI
if (process.env.ANTHROPIC_AUTH_TOKEN) { /* Anthropic 原生 */ }
else if (process.env.AI_GATEWAY_API_KEY) { /* 兼容网关 */ }
else if (process.env.OPENAI_API_KEY) { /* 官方 OpenAI */ }
```

### 2.4 考虑 Proxy 和模型不可用的情况

- **地区限制**：某些模型在某些云区域不可用（如返回 403），需要配置 fallback 模型
- **SOCKS5 代理**：通过环境变量配置，AI SDK 的 fetch 注入代理
- **Tool calling 不支持**：部分代理不支持原生 tool calling，需要降级为 prompt-based JSON 解析

```typescript

const cached = await db.aiCache.findUnique({ where: { hash } });
if (cached && !expired(cached)) return JSON.parse(cached.result);
```

## 四、从实际项目中提炼的 Insight

### 4.1 Prompt 版本管理是刚需

把 Prompt 存在数据库配置表（而非硬编码），可以：
- A/B 测试不同版本的 Prompt
- 按组织自定义 Prompt
- 版本回滚（新 Prompt 效果不好时切回旧的）

### 4.2 Structured Output 的 fallback

不能假设所有模型/代理都支持 `tool_calling`。代码要能同时处理「模型原生支持 structured output」和「模型只返回文本、需要手动解析 JSON」两种情况。`safeGenerateObject` 模式的本质就是自动降级。

### 4.3 「AI 建议 + 人工确认」是标准模式

除非有 99.9% 的准确率保证（没有人有），否则永远让 AI 结果可被人覆盖。用户对 AI 的修正（yes/no/edit）是最有价值的训练信号。

### 4.4 AI 不只是「聊天框」

把 AI 能力嵌入到业务流程的每个环节——分类、抽取、摘要、匹配、风险预测——而不是只做一个侧边栏聊天机器人。AI 是产品的一层能力，不是产品本身。
