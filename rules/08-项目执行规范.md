# 08 — 项目执行规范

> 作用：知道项目怎么分阶段执行、怎么和 AI 分工、怎么验收产出

---

## 一、人机分工

### 你做的（决策层）

- 定义产品定位和 MVP 范围
- 画出业务对象关系（ER 图）
- 定义 API 接口契约
- 画出页面布局和交互流程
- 决定技术选型
- Review AI 产出的代码
- 排查部署问题的根因
- Git 提交管理

### AI 做的（执行层）

- 生成 Prisma schema（你定义好关系后）
- 生成 API route 代码（你定义好接口后）
- 生成前端组件（你定义好页面后）
- 生成配置文件
- 写单元测试

### AI 做的（执行层 + 顾问层）

除执行外，AI 有权也有责任在发现问题时主动向你指出：

- 需求模糊时请你澄清
- 技术方案冲突时提出替代方案
- 设计有隐患时提醒你
- 有更简单的实现方式时建议你

详见 `rules/00-AI协作原则.md`。

### 核心原则

> 你做决策，AI 做执行 + 做顾问。你负责「做什么、为什么、对不对」，AI 负责「怎么写」和「告诉你可能踩什么坑」。
> 如果 AI 写的代码你看不懂，停下来先搞懂。积累多了项目就会变成你控制不了的黑箱。

## 二、项目四阶段执行法

### 第一阶段：骨架期（1-2 周）

**目标**：模板能跑起来

```
□ 从模板开始（不要从零写）
□ 理解数据库 schema（User, Organization, Member 等基础表）
□ 调整 auth 配置
□ 确认页面都能访问
□ CI/CD 能跑通
```

**不要做的事**：写任何业务代码

### 第二阶段：核心场景（2-4 周）

**目标**：一个场景完整跑通

```
□ 设计核心业务表 → prisma db push
□ 写核心业务逻辑（packages/ 中）
□ 实现 oRPC + MCP 双轨 API
□ 实现核心页面
□ 端到端验证：数据进入 → 处理 → 展示 → 用户操作 → 结果反馈
```

**原则**：只做一个场景，但要完整闭环

### 第三阶段：完善（2 周）

```
□ 第二个场景
□ 交互细节打磨
□ 错误处理完善
□ 权限细化
□ AI 功能增强
```

### 第四阶段：上线（1 周）

```
□ 生产环境部署
□ HTTPS / 域名
□ 监控和日志
□ 数据库备份
□ 文档
```

## 三、每阶段验收清单

每完成一个阶段，问这 6 个问题：

```
□ 有没有主对象？（核心实体是否清晰定义）
□ 有没有真实数据流？（前端→API→处理→落库→返回）
□ 有没有落库闭环？（关键动作有写数据库，不是 mock）
□ 有没有流程约束？（状态机在后端强制，不是前端随便改）
□ 有没有可追溯性？（audit log / timeline / createdBy / updatedAt）
□ 有没有扩展空间？（现在的实现不会阻碍下一阶段）
```

## 四、给 AI 的工作消息模板

### 模板 1：新增数据库 Model

发给 AI 的消息应该包含这三个部分：

```markdown
## 要做什么
在 Prisma schema 中添加 [对象名] Model

## 业务定义
- 属于 6 类对象中的哪一类：[主对象/过程对象/关系对象/审计对象/AI对象/配置对象]
- 字段和含义：
  field1: 类型, 含义, 必填/可选
  field2: ...
- 和其他表的关系：
  属于 Organization（一对多，外键 organizationId）
  关联到 User（通过 createdById）
- 索引需求：需要按 organizationId + status 组合查询

## 约束
- 遵循 rules/02-数据库设计规范.md
- createdAt / updatedAt 必须有
- 枚举用 enum 不用 string
- 长文本用 @db.Text
```

### 模板 2：实现一个功能 API

```markdown
## 要做什么
为 [功能名] 实现 oRPC API 和 MCP Tool

## API 契约
- list: params: organizationId(必填), status(可选), limit(可选,默认20)
         returns: { data: Item[], hasMore: boolean, nextCursor?: string }
- get:   params: id
         returns: 详情，包含关联数据
- create: body: { title, type, ... }
         returns: 创建后的完整记录
- update: body: { status?, ... }
         returns: 更新后的完整记录

## 约束
- 遵循 rules/03-API设计规范.md 和 rules/05-Tool与Agent架构规范.md
- 业务逻辑写在 packages/ 中
- oRPC procedure 和 MCP Tool 都只是适配层
- MCP Tool description 写清楚触发条件
```

### 模板 3：实现一个前端页面

```markdown
## 要做什么
实现 [页面名] 页面

## 页面布局
- 顶部：[搜索栏 + 筛选条件 + 操作按钮]
- 主体：[列表或卡片区域]
- 每个列表项显示：标题、状态标签、关键信息摘要、时间

## 交互流程
1. 进入页面 → loading skeleton
2. 数据加载 → 展示列表
3. 点击列表项 → 跳转详情页
4. 操作按钮 → 确认弹窗 → 执行 → toast → 自动刷新列表
5. 加载失败 → 错误提示 + 重试按钮
6. 列表为空 → 空状态引导

## 约束
- 遵循 rules/06-前端开发规范.md
- loading / error / empty 三种状态必须都有
- 数据用 TanStack Query
- 表单用 react-hook-form + zod
```

### 模板 4：排查问题

```markdown
## 问题描述
[具体现象：什么操作、期望什么、实际发生了什么]

## 已尝试
- [列出你已经做的排查]

## 相关上下文
- 相关文件的路径
- 最近改了什么
- 环境信息
```

## 五、从实际项目中提炼的 Insight

### 5.1 模板先行，业务后加

成熟的 AI SaaS 项目不是从零搭的，而是在模板基础上加业务代码。模板解决了基础设施问题（认证、数据库、支付、部署），你只需要写业务逻辑。

不要花时间重复造轮子。从模板开始，只关注业务。

### 5.2 一个场景要端到端跑通，而不是 5 个场景各做 60%

深度优先 > 广度优先。判断标准：
- 用户能不能完整走通一个任务？
- 数据流有没有断点？
- 有没有 mock 数据在蒙混过关？

### 5.3 产品工程师的核心能力

不是会写代码，而是：
1. **抽象能力**：看到业务需求能画出 ER 图
2. **拆解能力**：知道一个大需求要分几步做
3. **判断能力**：知道什么该做、什么不该做、什么时候停
4. **沟通能力**：能把需求说清楚，让 AI 准确执行

代码写得好不好是 AI 的事。结构设计、技术决策、产品判断是你的事。
