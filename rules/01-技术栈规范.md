# 01 — 技术栈规范

> 作用：确保所有项目使用一致的技术栈，AI agent 不随意引入新技术

---

## 一、硬性规则（MUST）

> ⚠️ 规则冲突提醒：如果以下规则与你的实际需求有任何冲突，AI 必须主动指出并建议替代方案（参见 00-AI协作原则.md）。

### 1.1 固定技术栈

| 层级 | 技术 | 不可替换为 |
|------|------|-----------|
| 框架 | Next.js 16 App Router | Nuxt / Remix |
| 语言 | TypeScript（strict mode） | JavaScript |
| 包管理 | pnpm + Turborepo | npm / yarn |
| 数据库 ORM | Prisma + PostgreSQL | TypeORM / Drizzle |
| 认证 | better-auth | NextAuth / Clerk |
| API（前端） | Hono + oRPC | tRPC / Express |
| API（Agent） | MCP (Model Context Protocol) | 自建 REST |
| UI | shadcn/ui + Radix + Tailwind | MUI / Ant Design |
| 表单 | react-hook-form + zod | Formik |
| 数据获取 | TanStack Query | useEffect / SWR |
| AI SDK | Vercel AI SDK v5 | 直接调 OpenAI SDK |
| 邮件 | React Email + Resend | Nodemailer |
| 支付 | Stripe | 其他 |
| 存储 | S3（MinIO 本地 / Cloudflare R2 生产） | 本地文件系统 |
| 代码质量 | Biome | ESLint + Prettier |

### 1.2 禁止引入的库

```
❌ 状态管理：Zustand / Redux / MobX（TanStack Query 足够）
❌ CSS 方案：CSS Modules / styled-components / CSS-in-JS
❌ 组件库：MUI / Ant Design / Chakra UI
❌ 表单工具：Formik
❌ 认证方案：NextAuth / Clerk（better-auth 已集成）
❌ ORM 切换：Drizzle（团队统一用 Prisma）
```

### 1.3 项目结构（Monorepo 规范）

```
项目根/
├── apps/
│   ├── web/              # 主应用（必须）
│   └── portal/           # 门户应用（可选，多端时加）
├── packages/
│   ├── database/         # Prisma schema + 数据库查询
│   ├── auth/             # better-auth 配置
│   ├── api/              # Hono + oRPC API 层
│   ├── ai/               # AI SDK 封装
│   ├── ui/               # shadcn/ui 组件库
│   ├── mail/             # 邮件发送
│   ├── payments/         # 支付
│   ├── storage/          # 文件存储
│   ├── i18n/             # 国际化
│   ├── utils/            # 通用工具函数
│   └── [业务包]/         # 👈 业务逻辑（按需创建）
├── tooling/              # 共享构建配置
├── docker-compose.yml    # 本地开发环境
└── rules/                # 本套规范文档
```

### 1.4 代码组织规则

```
MUST:     业务逻辑放在 packages/[业务包]/
MUST:     页面组件放在 apps/web/app/
MUST:     共享 UI 组件放在 packages/ui/
MUST:     API 路由在 packages/api/modules/ 中按功能分目录
MUST NOT: 在 apps/web 中直接写数据库查询
MUST NOT: 在 MCP Tool handler 中直接写业务逻辑
MUST NOT: 在组件中直接调用 Prisma client
```

## 二、强烈建议（SHOULD）

### 2.1 新增功能的正确顺序

```
1. packages/your-feature/actions.ts          ← 核心业务逻辑（从这里开始）
2. packages/api/modules/your-feature/        ← oRPC router（给前端 UI）
3. apps/web/app/.../your-feature/            ← 页面组件
4. （如果需要 Agent 调用）MCP Tool 注册      ← 给 AI Agent
```

❌ 错误顺序：先写页面 → 发现需要后端 → 在 API route 里堆业务逻辑

### 2.2 环境变量按用途分类

```bash
# 数据库
DATABASE_URL=

# 站点
NEXT_PUBLIC_SITE_URL=

# 认证
BETTER_AUTH_SECRET=
GOOGLE_CLIENT_ID=

# AI（支持多 Provider）
OPENAI_API_KEY=
ANTHROPIC_AUTH_TOKEN=
AI_TEXT_MODEL=      # 主力模型名称
AI_FAST_MODEL=      # 快速模型名称

# 存储
S3_*

# 邮件
RESEND_API_KEY=

# 支付
STRIPE_*

# 安全
FIELD_ENCRYPTION_KEY=  # 加密数据库中的敏感字段
CRON_SECRET=           # 定时任务鉴权
```

## 三、从实际项目中提炼的 Insight

### 3.1 双轨 API 是现代 AI 产品的标准模式

oRPC（给前端 UI 调用）+ MCP（给 AI Agent 调用）。核心业务逻辑写一次，两套接口只是薄薄的适配层。

不要只做 REST API 然后让 AI Agent 自己摸索调用方式。Agent 需要结构化的 Tool 描述（name, description, inputSchema），这是 MCP 协议的价值。

### 3.2 Monorepo 的包拆分会随业务增长演进

- 起步期：10 个基础包（database, auth, api, ai, ui, mail, payments, storage, i18n, utils）
- 成长期：拆分业务包（tools, workflows, automation）
- 成熟期：业务包进一步细分

**原则**：一开始不要拆太细。当单个包超过 20 个文件时再考虑拆分。

### 3.3 多端架构：一个 web app 不够时

如果产品有不同类型的使用者（如内部运营人员 vs 外部客户/合作伙伴），考虑用多个 Next.js app（`apps/web` + `apps/portal`）而不是在一个 app 里揉在一起。

这样做的好处：
- 路由冲突减少
- 部署可以独立（portal 可能用更少的资源）
- 安全边界更清晰（portal 用户不应该看到管理后台的路由）

### 3.4 AI SDK 需要支持多 Provider 切换

不要只绑死一个 AI 提供商。设计上支持：
- Anthropic 原生 API
- OpenAI 兼容网关（如 OpenRouter、OneAPI）
- 官方 OpenAI API

按优先级自动选择可用的 Provider。同时要考虑某些 Provider 不支持 `tool_calling` 的情况，提供 prompt-based JSON 解析的降级方案。
