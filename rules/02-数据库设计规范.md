# 02 — 数据库设计规范

> 作用：设计出正确的、可扩展的数据库结构

---

## 一、硬性规则（MUST）

> ⚠️ 规则冲突提醒：如果以下规则与你的实际需求有任何冲突，AI 必须主动指出并建议替代方案（参见 00-AI协作原则.md）。

### 1.1 每个表必须有这 2 个字段

```prisma
model YourModel {
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

### 1.2 命名规范

```
Model 名    PascalCase 单数（User, Organization, WorkflowInstance）
数据库表名  snake_case 复数（@map("users")）
字段名      camelCase
外键        [relatedModel]Id（organizationId, createdById）
枚举        PascalCase（EntityType, PurchaseType）
```

### 1.3 长文本必须 @db.Text

```prisma
model EmailRecord {
  bodyText  String? @db.Text
  bodyHtml  String? @db.Text
  content   String? @db.Text
}
```

### 1.4 外键必须显式声明 onDelete

```prisma
model Member {
  organization   Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  // 想清楚：删组织 → 成员也删 → Cascade
  //         删用户 → 帖子保留 → SetNull
  //         删用户 → 已关联订单阻止删除 → Restrict
}
```

### 1.5 AI 输出必须用 Zod Schema 约束

```typescript
// ✅ 正确
const ClassificationSchema = z.object({
  type: z.enum(["approval", "report", "issue"]),
  confidence: z.number().min(0).max(1),
  reasoning: z.string().max(100),
});

const result = await generateObject({ model, schema: ClassificationSchema, prompt });

// ❌ 错误：只在 prompt 里描述 JSON 格式，然后手动 parse
```

## 二、强烈建议（SHOULD）

### 2.1 业务对象分 6 类建模

| 对象类型 | 特征 | 例子 |
|---------|------|------|
| **主对象** | 业务核心，独立存在 | 订单、项目、签约 |
| **过程对象** | 记录动作，带时间戳 | 审批记录、邮件线程、访问日志 |
| **关系对象** | 多对多关联 | 标签关联、人选匹配 |
| **审计对象** | 谁在什么时候做了什么 | 变更日志、操作日志 |
| **AI 对象** | LLM 产出的结构化结果 | AI 评估、记忆条目 |
| **配置对象** | 系统配置，不参与业务流程 | 分类规则、工作流模板 |

### 2.2 JSON 字段 vs 关联表的选择

**用 JSON 字段（`Json` 或 `@db.Text`）：**
- 字段结构频繁变化（如 AI 提取结果每次迭代都可能改字段）
- 不需要单独查询 JSON 内部的值
- 永远跟着主记录一起出现

**用关联表：**
- 需要单独查询（"所有标签为 X 的帖子"）
- 需要被其他表引用
- 有独立的增删改查生命周期

**典型场景**：AI 提取的结构化字段用 JSON 存（因为不同分类提取的字段完全不同，且随 Prompt 迭代频繁变化），而分类-标签的多对多关系用中间表。

### 2.3 索引建在这些地方

```prisma
// WHERE 条件里经常出现的字段
@@index([organizationId])
@@index([organizationId, type])
@@index([organizationId, status])

// 外键字段
@@index([userId])
@@index([entityId])

// 排序字段 + 筛选字段的组合
@@index([organizationId, createdAt])

// 不要对所有字段建索引：
// - 区分度很低的字段（如只有 3 个值的 status）
// - 频繁写入但很少读取的字段
```

### 2.4 用枚举约束状态而不是 String

```prisma
// ✅ 正确
enum EntityType { APPROVAL; REPORT; ISSUE }
model Entity { type EntityType }

// ❌ 错误：什么都写不进去，typo 不会被发现
model Entity { type String }
```

## 三、建议（MAY）

### 3.1 软删除 > 硬删除

```prisma
model Entity {
  deletedAt DateTime?  // null = 未删，有值 = 已删除
}
```

### 3.2 关键业务流程存变更历史

不要只存当前状态（如 `status: "approved"`），要存完整的状态变更记录：

```prisma
model StageTransitionLog {
  entityId   String
  fromStage  String    // 变更前
  toStage    String    // 变更后
  changedBy  String    // 谁改的
  reason     String?   // 为什么改
  createdAt  DateTime
}
```

这比单纯打 `updatedAt` 强得多——不只是知道「被改过了」，还知道「谁在什么时候从什么改成了什么、为什么」。

### 3.3 考虑全栈类型链路

```
Prisma Schema → Prisma Zod Generator → Zod Schema → oRPC types → TanStack Query → 前端类型
```

改一个字段，整条链路类型报错，不会遗漏任何地方。

## 四、从实际项目中提炼的 Insight

### 4.1 AI 产品的 Schema 要有弹性

不要把 AI 输出结构写死在表结构里。审批邮件的提取字段（金额、申请人、部门）和报告邮件的提取字段（进展、阻塞项、下步计划）完全不同，而且会随 Prompt 迭代变化。用 JSON 字段存 AI 提取结果，让 Schema 保持灵活。

### 4.2 大规模项目的模型组织

实际项目可能有 100+ 个 Model，但每个都能归入 6 类对象。面对复杂数据模型时，先分类再逐个理解：

- 主对象的数量 ≈ 核心业务领域的数量
- 复杂程度 = 主对象数量 × 每个主对象的状态机复杂度
- 审计对象的完善程度 ≈ 企业级成熟度

### 4.3 多租户设计

每个业务表都有 `organizationId`，每个查询都带 `WHERE organizationId = ?`。这不是多余的——是实现数据隔离最简单可靠的方式。不要在应用层做隔离。

### 4.4 Memory/对象模型设计模式

如果需要 AI 记忆功能，用双表设计：

```
MemoryObject（结构化知识卡）：类型 + ID + 属性(JSON) + 关系(JSON)
  ↓ 1:N
MemoryEntry（叙事型日志）：分类 + 内容 + 重要性 + 向量嵌入

Object 存「当前状态」，Entry 存「变更历史 + 叙事片段」
```

这种设计让 AI 既能读到结构化的当前画像（快），又能检索到历史上下文（全）。
